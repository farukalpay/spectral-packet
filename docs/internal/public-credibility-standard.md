# Public Credibility Standard

All public-facing repository surfaces must satisfy this rule:

The repository must look like one coherent scientific compute product whose engine depth is visible before its extension breadth.

That requires:

- immediate clarity about what the product is and is not,
- a curated first-run path that reaches a real artifact quickly,
- visible implementation depth in the core engine and its tests,
- stable, reference, and experimental surfaces labeled explicitly,
- interface layers that read as thin wrappers over shared workflows,
- no inflated claims and no roadmap language masquerading as implementation,
- no junk, residue, or internal-looking files in the public-facing tree,
- public examples that assume an installed library instead of local path tricks.

If a change improves capability but harms first impression, structural coherence, or trust, it does not meet the public credibility standard.
